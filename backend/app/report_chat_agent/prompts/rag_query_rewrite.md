## 改写原则
1. 不要只复制用户原问题，必须融合当前报告的业务锚点（场景、异常项、风险项、建议项、指标名称）。
2. 消解指代：将"这个""该异常""类似情况""这种风险"替换为具体的异常/风险名称。
3. query 必须是包含完整业务语义的检索语句，不需要带问号。
4. 不编造报告中不存在的数值、站点、设备、部门名称。
5. 不扩大查询范围：不加入"全公司""全部""全业务域""所有时间"等无限制范围词。
6. RAG 查询的是知识依据（制度、标准、案例、经验），不是实时业务数据。query 中不应出现趋势、统计、实时状态类词汇。
7. 长度控制在 15~50 个中文字符，过短的 query 检索意图不明确，过长的 query 引入噪声。
8. 如果用户问题明确指向特定制度/标准（如《本质安全隐患治理规范》），query 中应包含完整文档名称。

## 输出要求
仅输出 JSON，不要包含任何其他内容：

```json
{
  "query": "本质安全场景下，超期未处理缺陷的处置要求和整改流程",
  "query_keywords": ["超期未处理缺陷", "处置要求", "整改流程"],
  "doc_types": ["制度", "标准"],
  "reason": "替换指代并融合场景和异常项锚点的简短说明",
  "confidence": 0.88
}
```

### 字段说明
- query: 改写后的 RAG 检索查询文本，15~50 个中文字符
- query_keywords: 从 query 中提取的核心检索关键词，3~5 个
- doc_types: 建议查询的文档类型列表：制度、标准、案例、经验
- reason: 改写原因的简短说明
- confidence: 改写置信度，0.0~1.0
