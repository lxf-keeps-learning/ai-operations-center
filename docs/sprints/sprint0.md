## 3. 当前项目目录理解

当前采用一个总仓库管理前后端，前后端通过不同目录区分：

```text
AIOperationsCenter/
├── frontend/       # 前端工程
├── backend/        # 后端工程
├── docs/           # 项目文档
├── scripts/        # 脚本目录
├── deploy/         # 部署配置预留
├── README.md       # 项目说明
└── .env.example    # 环境变量模板
```

这种结构的核心理解是：

```text
一个仓库统一管理，不代表一个服务统一启动。
```

实际启动时：

- 前端独立启动。
- 后端独立启动。
- MySQL 独立运行。

前后端放在一个总目录下，是为了统一管理、统一文档、方便联调；前后端服务分开启动，是因为它们属于不同技术栈和不同运行进程。

## 5. 前后端分离理解

本项目采用前后端分离模式。

前端职责：

- 页面展示。
- 用户交互。
- 数据展示。
- 调用后端 API。

后端职责：

- 提供 API 接口。
- 处理业务逻辑。
- 连接数据库。
- 后续封装 AI Agent、LLM Client、Tool 调用等能力。

前后端分离带来的收益：

1. **职责明确**  
   前端专注页面和交互，后端专注接口、数据和业务逻辑。

2. **技术栈独立**  
   前端可以使用 Vue3，后端可以使用 FastAPI，双方通过 API 协作。

3. **便于团队协作**  
   前端开发和后端开发可以并行推进，只要接口约定清晰即可。

4. **便于独立部署**  
   后续前端可以单独构建部署，后端可以独立作为 API 服务部署。

5. **便于系统扩展**  
   后续接入 AI Agent、多个业务系统、多个数据源时，后端可以持续扩展能力，前端只负责展示结果和交互入口。

## 7. 为什么建立 Git 规范

Git 规范是企业项目协作的基础。

一个项目可能由多人同时开发，如果没有分支规范和提交规范，容易出现：

- 代码互相覆盖。
- 主分支不稳定。
- Bug 难以回溯。
- 合并冲突难以处理。
- 无法清楚知道每次提交改了什么。

因此需要建立：

- 分支规范。
- Commit 规范。
- Merge 规范。
- Review 机制。

Git 规范不是形式主义，而是保证项目长期稳定迭代的重要基础设施。

## 8. 关键架构决策复盘

### 8.1 为什么企业项目先做工程初始化，而不是先写业务？

工程初始化主要完成 Git 仓库、前后端工程、目录结构、环境配置、数据库连接、接口验证、文档说明等基础能力。

只有在项目启动阶段把这些基础能力建设好，整体项目架构才会清晰，后续开发才更便于管理、维护和扩展。

如果一上来直接写业务，容易导致层级不清晰、代码混乱、重复代码增多、配置散落，后续重构成本更高。

---

### 8.2 前后端分离带来的收益是什么？如何支撑团队协作？

前后端分离的核心收益是职责明确。

前端负责页面和交互，后端负责接口、业务、数据库和 AI 能力。双方通过 API 进行通信。

在团队协作上，可以按照功能模块创建不同的开发分支，前后端人员并行开发。开发完成后通过 Pull Request 或 Merge Request 申请合并，经过测试和 Review 后再合并到集成分支。

这种方式可以降低耦合，提高协作效率，也便于后续独立部署。

---

### 8.3 这套目录后续如何扩展 Agent？

当前目录为后续 Agent 扩展预留了基础空间。

后端可以逐步扩展为：

```text
backend/app/
├── api/          # 对外接口
├── services/     # 业务服务
├── models/       # 数据库模型
├── schemas/      # 请求与响应结构
├── agents/       # Agent 编排能力
├── graphs/       # LangGraph 或 Graph 编排
├── tools/        # Agent 可调用工具
└── core/         # 配置、日志、异常处理等基础能力
```

后续不同业务系统可以对应不同 Agent 或不同 Subgraph，由 MainGraph 统一编排和调度。

例如：

```text
MainGraph

```

前端可以按照业务场景扩展不同页面和组件，例如：

```text
frontend/src/

│   ├── alarm-analysis/
│   ├── operation-dashboard/
│   └── report-center/
├── components/

```

### 8.4 后续接入多个业务系统时目录如何扩展？

多个业务系统接入时，前端可以按照业务场景拆分页面和组件；后端可以按照领域拆分 service、agent、tool 和 router。

推荐思路：

- 公共能力放到 common/core/shared 目录。
- 业务专属能力放到对应业务模块。
- Agent 能力按场景拆分，不要全部堆到一个文件里。
- Tool 能力要可复用，避免和具体页面强绑定。

后续可以形成：

```text
backend/app/
├── modules/
│   ├── alarm/
│   ├── energy/
│   ├── report/
│   └── operation/
├── agents/
├── tools/
└── graphs/
```

这样可以避免随着业务复杂度上升，代码变得冗余和不可维护。

---

### 8.5 为什么 AI Runtime 不直接放进原运营中心系统？

原运营中心系统已经能够完成数据展示闭环，不适合直接在原系统中强行嵌入复杂 AI Runtime。

主要原因：

1. **技术栈差异**  
   原前端可能是 Vue2，当前 AI 能力建设更适合使用 Vue3 + TypeScript + Vite，以获得更好的开发体验和扩展能力。

2. **后端职责差异**  
   原后端是 Java 服务，系统较重，主要承担已有业务闭环。AI Agent 更适合使用 Python + FastAPI 构建独立服务，便于模型调用、Agent 编排、工具封装和异步任务扩展。

3. **降低对原系统影响**  
   AI 能力是对原运营中心的增强，而不是重构原系统。独立建设 AI 能力层，可以降低对原系统的侵入和风险。

4. **便于独立演进**  
   AI Agent 后续可能涉及 LangGraph、LLM Client、Tool、任务调度、缓存、日志、模型调用等能力，独立服务更便于演进。

一句话总结：

```text
AI Runtime 是对原运营中心的能力增强，不是对原系统的推翻重建。
```

# Sprint0 面试题与参考答案

## 1. 什么是 Sprint0？

Sprint0 是项目正式进入业务开发前的启动阶段，主要完成工程初始化、目录结构、技术选型、运行环境、Git 规范、基础联调和文档沉淀。

在本项目中，Sprint0 完成了前端 Vue3 工程、后端 FastAPI 工程、MySQL 数据库连接、Demo CRUD、前后端通信和基础工程规范。

---

## 2. 为什么企业项目不建议一上来直接写业务代码？

因为企业项目需要长期维护和多人协作。如果没有先完成工程初始化，后续容易出现目录混乱、配置混乱、代码冗余、接口难联调、部署困难等问题。

先完成工程初始化，可以让后续业务开发建立在稳定、清晰、可扩展的基础上。

---

## 3. 前后端分离的收益是什么？

前后端分离可以让前端专注页面展示和用户交互，后端专注接口、业务逻辑、数据库和 AI 能力。

主要收益包括：

- 职责清晰。
- 技术栈独立。
- 便于团队协作。
- 便于独立部署。
- 便于后续扩展。

---

## 4. 为什么前后端在一个项目目录下，但需要分别启动？

因为一个仓库管理不等于一个服务启动。

前端是 Node/Vite 服务，后端是 Python/FastAPI 服务，两者属于不同运行进程，因此需要分别启动。

把它们放在一个总仓库下，是为了统一管理、统一文档和方便联调。

---

## 5. FastAPI 的 `/docs` 有什么作用？

`/docs` 是 FastAPI 自动生成的 Swagger 接口文档，可以查看接口路径、请求参数、响应结构，并可以直接在线测试接口。

在 Sprint0 中，`/docs` 是验证后端服务和接口是否正常的重要入口。

---

## 6. MySQL 在项目中承担什么职责？

MySQL 是项目的主数据库，用于持久化存储数据。

在 Sprint0 阶段，MySQL 主要用于验证后端数据库连接和 Demo CRUD。后续可以存储用户操作、Agent 任务、AI 调用日志、分析结果、系统配置等数据。

---

## 7. Navicat 是数据库吗？

不是。Navicat 是数据库可视化管理工具，MySQL 才是真正的数据库服务。

Navicat 的作用是连接数据库，方便查看表结构、数据记录和执行 SQL。

---

## 8. 什么是 `DATABASE_URL`？

`DATABASE_URL` 是后端连接数据库的统一配置字符串。

示例：

```env
DATABASE_URL=mysql+pymysql://<username>:<password>@localhost:3306/ioc_ai
```

它包含数据库类型、驱动、用户名、密码、主机、端口和数据库名。

---

## 9. 为什么不建议项目长期使用 root 用户连接数据库？

因为 root 用户权限过高，不符合最小权限原则。

企业项目中应为具体项目创建独立数据库用户，只授予该项目数据库的必要权限，这样更加安全，也便于权限管理和问题排查。

---

## 10. `created_at` 和 `updated_at` 字段有什么作用？

`created_at` 表示数据创建时间，`updated_at` 表示数据最后更新时间。

它们常用于数据审计、问题排查、排序、同步和生命周期管理。

---

## 11. 后端为什么要分 router、service、model、schema？

这是为了让代码职责清晰。

- router：负责 HTTP 请求和响应。
- service：负责业务逻辑。
- model：负责数据库表结构映射。
- schema：负责请求和响应数据结构。
- db：负责数据库连接。

如果所有逻辑都写在 router 中，后续会很难维护。

---

## 12. SQLAlchemy 是什么？

SQLAlchemy 是 Python 中常用的 ORM 工具，用于通过 Python 对象操作数据库。

它可以把数据库表映射为 Python 类，让开发者通过对象方式完成新增、查询、更新和删除。

---

## 13. ORM 有什么好处？

ORM 的好处包括：

- 减少重复 SQL。
- 提高代码可读性。
- 让表结构和代码模型对应。
- 便于维护数据库访问逻辑。
- 一定程度上支持数据库切换。

但复杂查询和性能优化场景下，仍然需要理解 SQL。

---

## 14. 为什么接口路径建议统一加 `/api` 前缀？

统一 `/api` 前缀可以区分前端页面路由和后端接口路由，也方便后续做网关、代理、版本管理和权限控制。

例如：

```text
/api/health
/api/items
/api/ai/chat
```

---

## 15. 什么是健康检查接口？

健康检查接口用于判断服务是否正常运行。

例如：

```text
GET /api/health
```

它可以用于前后端联调、部署检查、服务监控和问题排查。

---

## 16. Vue3 + Vite + TypeScript 的组合有什么优势？

Vue3 适合构建现代前端页面，Vite 启动速度快、开发体验好，TypeScript 可以提供类型检查，减少代码错误。

这个组合适合快速搭建前端工程，并与后端 API 服务联调。

---

## 17. 前端如何调用后端接口？

前端可以通过 `fetch` 或 `axios` 调用后端接口。

本地开发时可以直接调用：

```text
http://127.0.0.1:8000/api/items
```

也可以通过 Vite proxy 转发，避免跨域问题。

---

## 18. 什么是跨域问题？

当前端页面地址和后端接口地址的协议、域名或端口不一致时，就可能出现跨域问题。

例如：

```text
前端：http://localhost:5173
后端：http://127.0.0.1:8000
```

可以通过后端配置 CORS 或前端配置代理解决。

---

## 19. Redis 在项目中有什么作用？

Redis 不是主数据库，而是缓存和临时状态存储组件。

在 AI Agent 项目中，Redis 后续可用于：

- Agent 任务状态缓存。
- 会话缓存。
- 接口限流。
- 流式输出状态。
- 临时结果缓存。

---

## 20. 为什么 Sprint0 没有强制接入 Redis？

Redis 是后续增强能力，不应该在项目启动阶段阻塞主流程。因此本阶段先预留 Redis，后续在任务状态、会话、缓存场景中再正式接入。

## 21. Docker Compose 的作用是什么？

Docker Compose 用于统一编排本地依赖服务，例如 MySQL、PostgreSQL、Redis、后端服务等。

它可以解决环境不一致、新人启动困难、服务依赖复杂等问题。

当前本地未安装 Docker，因此 Sprint0 暂未强制依赖 Docker，后续可补齐 Docker Compose 标准模式。
